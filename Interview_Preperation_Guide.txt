Java 8 Features – Interview Questions & Answers

Q: What are the main features introduced in Java 8?
A: Lambda Expressions, Functional Interfaces, Streams API, Default and Static Methods in Interfaces,
Optional Class, Date and Time API, and Nashorn Engine.
Q: What is a Lambda Expression?
A: A short block of code that takes parameters and returns a value. Enables functional programming and
replaces anonymous classes.
Q: What is a Functional Interface?
A: An interface with exactly one abstract method. Examples include Runnable, Callable, Comparator. Can
be annotated with @FunctionalInterface.
Q: What are Default and Static Methods in Interfaces?
A: Default methods use the 'default' keyword, allowing backward compatibility. Static methods belong to the
interface itself.
Q: What is the Streams API and why is it used?
A: Used for processing collections in a functional style using operations like filter, map, and reduce.
Q: What is the difference between map() and flatMap()?
A: map() transforms each element, while flatMap() flattens nested structures into a single stream.



#######################################################################################################################################################################################



Q: What is the Optional class and why is it used?
A: To handle null values safely and avoid NullPointerExceptions. It represents optional values explicitly.
Q: What is the difference between forEach() and map()?
A: forEach() performs an action on elements, map() transforms elements and returns a new stream.
Q: How can you filter and collect data using Streams?
A: By chaining filter() and collect() methods. Example: names.stream().filter(n ->
n.startsWith('V')).collect(Collectors.toList()).
Q: Explain Predicate, Function, Consumer, and Supplier interfaces.
A: Predicate: returns boolean; Function: transforms data; Consumer: processes data without returning;
Supplier: provides data without input.
Q: What is the difference between Stream and Collection?
A: Collection stores data, Stream processes data. Streams are computed on demand and not reusable.
Q: Can Streams be reused?
A: No. Once a terminal operation is executed, the stream is closed and cannot be reused.
Q: What are Intermediate and Terminal Operations in Streams?
A: Intermediate: filter, map, sorted; Terminal: collect, forEach, count.
Q: What is the difference between findFirst() and findAny()?
A: findFirst() returns the first element, findAny() returns any element and is optimized for parallel streams.
Q: How can you create a Stream in Java 8?
A: Using Stream.of(), Arrays.asList().stream(), or Stream.iterate().limit()

    Steps:
    1. Calculate distanceA = abs(liftA_floor - user_floor)
    2. Calculate distanceB = abs(liftB_floor - user_floor)

    3. If distanceA < distanceB:
           return "Lift A"
       Else if distanceB < distanceA:
           return "Lift B"
       Else:
           // Both lifts are equidistant
           // Apply tie-breaker rule
           If liftA_idle_time > liftB_idle_time:
               return "Lift A"
           Else if liftB_idle_time > liftA_idle_time:
               return "Lift B"

⭐ What is Stream API?

Stream API is a feature introduced in Java 8 that allows processing of collections in a declarative and functional style using operations like filter, map, and reduce.
It provides a pipeline of operations where intermediate operations are lazy and terminal operations produce the final result.

⭐ Important Properties

Not a data structure → doesn’t store elements
Functional → uses lambda expressions
Lazy evaluation → intermediate operations run only when terminal op is called
Can be parallel → parallelStream()
Doesn’t modify original collection

⭐ Types of Operations

✔ Intermediate → returns Stream
filter(), map(), sorted(), limit(), skip(), distinct()

✔ Terminal → ends stream
forEach(), collect(), reduce(), count(), anyMatch(), findFirst()

⭐ Pipeline Example
collection.stream() → filter() → map() → collect()

✅ ArrayList vs LinkedList (Theory + Scenario)

| Feature                    | ArrayList                | LinkedList                         |
| -------------------------- | ------------------------ | ---------------------------------- |
| **Underlying DS**          | Dynamic array            | Doubly linked list                 |
| **Access/Indexing**        | Very fast (O(1))         | Slow (O(n))                        |
| **Insert/Delete (middle)** | Slow (shifting elements) | Fast (just pointer update)         |
| **Memory**                 | Less memory              | More memory (prev & next pointers) |


Scenario Questions

Scenario 1: You need to retrieve elements frequently. Which one?
👉 ArrayList, because random access is O(1).

Scenario 2: You need to insert/delete many elements in the middle. Which?
👉 LinkedList, because no shifting needed.

Scenario 3: Add at end frequently?
Both work, but ArrayList is generally preferred due to cache locality.

✅ Overloading vs Overriding (Theory + Scenario)

Overloading

Same method name
Different parameter list
Happens at compile time (static binding)

Overriding

Same method signature in parent & child
Happens at runtime (dynamic binding)
Uses polymorphism (runtime behavior)

Scenario Questions

Scenario 1: Logging framework has multiple log() methods accepting String, int, Object. What concept?
👉 Overloading.

Scenario 2: Child class changes logic of parent class method. What?
👉 Overriding.

Scenario 3: When does JVM decide which overridden method to call?
👉 At runtime, based on the actual object (dynamic dispatch).

✅ Optional.of() vs Optional.ofNullable() vs Optional.empty()

Optional.of(value)

Throws NullPointerException if value is null
Use when you are sure the value is NON-NULL.

Optional.ofNullable(value)

Accepts both null and non-null
Null → Optional.empty()

Optional.empty()

Returns an empty Optional
Used when you want to explicitly return no value

Scenario

Scenario: You fetch user email from DB which might be null. Which one?
👉 ofNullable(email)

Scenario: You want to guarantee mandatory fields (e.g., id).
👉 Optional.of(id)

public static int[] twoSum(int[] nums, int target) {
    Map<Integer,Integer> map = new HashMap<>();
    for (int i=0;i<nums.length;i++){
        int need = target - nums[i];
        if (map.containsKey(need)) return new int[]{map.get(need), i};
        map.put(nums[i], i);
    }
    return new int[]{};
}

11) Balanced Parentheses (stack)
public static boolean isBalanced(String s){
    Deque<Character> st = new ArrayDeque<>();
    for (char c: s.toCharArray()){
        if (c=='(' || c=='{' || c=='[') st.push(c);
        else {
            if (st.isEmpty()) return false;
            char p = st.pop();
            if ((c==')' && p!='(') || (c=='}' && p!='{') || (c==']' && p!='[')) return false;

    return st.isEmpty();
}

12) Merge Two Sorted Arrays (merge step)
public static int[] mergeSorted(int[] a, int[] b){
    int i=0,j=0,k=0;
    int[] res = new int[a.length + b.length];
    while(i<a.length && j<b.length) res[k++] = (a[i]<=b[j]) ? a[i++] : b[j++];
    while(i<a.length) res[k++] = a[i++];
    while(j<b.length) res[k++] = b[j++];
    return res;
}

29) Remove Loop from LinkedList

Find meeting point then start of loop:

public static void removeLoop(ListNode head){
    ListNode slow=head, fast=head;
    boolean found=false;
    while (fast!=null && fast.next!=null){
        slow=slow.next; fast=fast.next.next;
        if (slow==fast){ found=true; break; }
    }
    if (!found) return;
    slow = head;
    if (slow==fast) {
        while (fast.next != slow) fast = fast.next;
    } else {
        while (slow.next != fast.next){ slow = slow.next; fast = fast.next; }
    }
    fast.next = null;
}

32) Sieve of Eratosthenes — All primes up to N
public static List<Integer> sieve(int n){
    boolean[] isPrime = new boolean[n+1];
    Arrays.fill(isPrime, true);
    isPrime[0]=isPrime[1]=false;
    for (int p=2;p*p<=n;p++) if (isPrime[p]){
        for (int q=p*p;q<=n;q+=p) isPrime[q]=false;
    }
    List<Integer> res = new ArrayList<>();
    for (int i=2;i<=n;i++) if (isPrime[i]) res.add(i);
    return res;
}

34) Bubble / Selection / Insertion Sort (brief)

Bubble:

public static void bubbleSort(int[] a){
    int n=a.length;
    for (int i=0;i<n-1;i++){
        boolean swapped=false;
        for (int j=0;j<n-1-i;j++){
            if (a[j] > a[j+1]) { int t=a[j]; a[j]=a[j+1]; a[j+1]=t; swapped=true; }
        }
        if (!swapped) break;
    }
}


Selection:

public static void selectionSort(int[] a){
    for (int i=0;i<a.length;i++){
        int min=i;
        for (int j=i+1;j<a.length;j++) if (a[j]<a[min]) min=j;
        int t=a[i]; a[i]=a[min]; a[min]=t;
    }
}


Insertion:

public static void insertionSort(int[] a){
    for (int i=1;i<a.length;i++){
        int key = a[i], j=i-1;
        while (j>=0 && a[j] > key){ a[j+1]=a[j]; j--; }
        a[j+1]=key;
    }
}


All are O(n^2) worst-case.

35) Kadane’s Algorithm — max subarray sum
public static int kadane(int[] a){
    int maxSoFar = a[0], maxEnding = a[0];
    for (int i=1;i<a.length;i++){
        maxEnding = Math.max(a[i], maxEnding + a[i]);
        maxSoFar = Math.max(maxSoFar, maxEnding);
    }
    return maxSoFar;
}
// Time O(n)

Here’s a clear, interview-friendly explanation of Circuit Breaker (Resilience4j) with examples and how it works in Spring Boot.

✅ What is a Circuit Breaker (Resilience4j)?

A Circuit Breaker is a resilience pattern used in microservices to prevent continuous calls to a failing downstream service.
It helps your system fail fast, recover gracefully, and avoid cascading failures.

Resilience4j is a lightweight Java library that provides fault-tolerance patterns such as:

CircuitBreaker

Retry

RateLimiter

Bulkhead

TimeLimiter

🎯 Why do we need a Circuit Breaker?

Imagine Service A → calls Service B.
If Service B becomes slow or goes down:

Service A will also become slow

Thread pools will fill up

Entire system may crash

Circuit Breaker stops calling Service B after detecting failures and gives a fallback response instead.

🔌 Circuit Breaker States
4
1️⃣ Closed (Normal State)

Calls flow normally

Failures are counted

If failure rate reaches threshold → Opens.

2️⃣ Open (Failure State)

No calls are allowed to downstream

Immediately returns fallback

Gives the system time to recover

After waitDuration → moves to Half-Open.

3️⃣ Half-Open (Testing State)

Allows limited trial calls

If they succeed → back to Closed

If they fail → back to Open

⚙️ Resilience4j Circuit Breaker in Spring Boot
@CircuitBreaker(name = "productService", fallbackMethod = "fallbackProduct")
public Product getProduct(String id) {
    return webClient.get()
        .uri("http://inventory-service/products/" + id)
        .retrieve()
        .bodyToMono(Product.class)
        .block();
}

public Product fallbackProduct(String id, Throwable ex) {
    return new Product(id, "Default Product", 0);
}

🛠️ Configuration (application.yml)
resilience4j:
  circuitbreaker:
    instances:
      productService:
        failure-rate-threshold: 50       # % of failures to open the CB
        slow-call-rate-threshold: 50
        slow-call-duration-threshold: 2s
        minimum-number-of-calls: 10      # evaluate after 10 calls
        wait-duration-in-open-state: 5s
        permitted-number-of-calls-in-half-open-state: 3

📌 Example Scenario
Service B Status	Circuit Breaker State	Behavior
Responding fine	Closed	Calls allowed
Slow or failing 50%+	Open	Immediate fallback
After 5 seconds	Half-Open	Allows few test calls
Test calls successful	Closed	Normal calls continue
🧠 Simple Analogy

Circuit breaker works like a home electrical fuse:

Too much load? → Fuse trips → Prevents damage

After some time → You try turning it on again

Same logic, but for microservice calls.

📌 When to Use Circuit Breaker?

Use it when:

✔ Calling unreliable external APIs
✔ Calling slow downstream microservices
✔ Calling databases or ML services with unpredictable latency

Avoid when:

❌ The call is very fast and always reliable
❌ Overhead of monitoring is unnecessary

✅ What is DB Indexing?

Database Indexing is a technique used to speed up data retrieval from a table.

An index works like the index of a book:

Without index: you scan every page → slow (full table scan)

With index: you jump directly to the page → fast (indexed lookup)

So indexing improves SELECT query performance.

🔍 Visual Understanding
4

Most databases (MySQL, Postgres, Oracle) internally use a B-Tree index, which allows logarithmic search time (O(log n)) instead of scanning entire rows.

🧠 Why do we need an index?

Without an index, the database must check each row one by one:

SELECT * FROM employees WHERE emp_id = 101;


If emp_id is not indexed, DB will scan the whole table — slow for large data.

If emp_id is indexed, DB jumps directly to the matching record — much faster.

📌 Where are indexes used?

Indexes help with columns used in:

WHERE conditions

JOIN conditions

ORDER BY / GROUP BY

Searching large tables

Foreign keys and primary keys

Example:

CREATE INDEX idx_employee_name ON employees(name);

⚙️ Types of Indexes
1️⃣ Primary Key Index

Automatically created for primary key → unique, fast lookups.

2️⃣ Unique Index

Guarantees no duplicate values.

3️⃣ Non-unique (Normal) Index

Speeds up searches but allows duplicates.

4️⃣ Composite Index (multiple columns)

Example:

CREATE INDEX idx_emp_name_dept ON employees(name, department);

5️⃣ Full-text Index

Used for text search (e.g., articles, blog search).

📉 Downside of Indexing

Indexes improve read speed, but slow down writes:

INSERT

UPDATE

DELETE

Because DB must update the index also.

So too many indexes = slow writes + heavy storage usage.

⭐ Interview-Ready One-Line Answer

DB indexing is a data structure used by databases to speed up searching and querying by allowing faster lookups, similar to an index in a book.

🔥 Real-time example (Microservices / Backend)

If you have an orders table with 10 million rows:

If you frequently search by order_id or customer_id, indexing those columns can reduce query time from seconds to milliseconds.

Performing the same operation multiple times has the same effect as performing it once.

Redis (Remote Dictionary Server) is an open-source, in-memory key–value store known for very high performance. Because data is stored in RAM, Redis is extremely fast (sub-millisecond latency).

To reduce database load and speed up responses.

Used in distributed systems for centralized session management.

3. Message queues / pub-sub

Redis has built-in Pub/Sub, Streams, and Lists so it can act like a lightweight message broker.

4. Rate limiting

API rate limiting, login attempt throttling, OTP sending limit, etc.

5. Distributed locks

To safely handle shared resources in microservices.

6. Leaderboard counters

Because Redis supports atomic increments (INCR, DECR).

HyperLogLog

AOF	Append-only log (writes every command)

Great question — this is the MOST important part of JWT security.

Let’s understand what happens if someone steals (fishes/phishes) a JWT token and how real systems protect against it.

🛑 If someone steals your JWT token — they can act as you

Because JWT is:

stateless

self-contained

Whoever has the token → is considered the user until token expires.

BUT…
There are STRONG ways to prevent and handle this.

🔐 1. Short Expiry Access Token (Best Practice)

Make JWT token valid for 5–15 minutes only.

So even if hacker steals it:
→ They can use it only for few minutes.

Example:

exp: now + 10 minutes

🔑 2. Use Refresh Tokens (Secure)

Flow:

Access token = short validity

Refresh token = long validity (7–30 days)

Refresh token is stored in HttpOnly Cookie (cannot be read by JavaScript → avoids XSS)

If access token is stolen → its lifetime is very short.

If refresh token is stolen → backend can detect misuse (see next point).

🛑 3. Store Refresh Tokens in Database → Allow Revocation

This is how 99% real systems handle token theft.

When user logs in, you store:

userId | refreshToken | deviceInfo | expiration


If suspicious activity happens:

User logs out

Token is used from strange device

Too many refresh attempts from different IPs

You mark token as revoked.

Then hacker’s refresh token becomes useless.

🕵️ 4. Rotate Refresh Tokens (Highly Secure)

Every time user uses refresh token:

Backend generates a new refresh token

Invalidates the old one

So if a hacker steals an old token →
it becomes invalid as soon as user refreshes.

This prevents replay attacks.

🌐 5. Use HTTPS ALWAYS

If your API uses HTTP, then:

Token travels in plain text

Anyone on same network can read it

Using tools like Wireshark

With HTTPS, tokens are encrypted during transmission.

🛡️ 6. Use HttpOnly Cookies instead of localStorage

If JWT is stored in localStorage:
→ It can be stolen via XSS attack

Use HttpOnly + Secure cookie, which JavaScript cannot access.

Set-Cookie: jwt=xxx; HttpOnly; Secure; SameSite=Strict;

👁️‍🗨️ 7. Device/IP Based Detection

Backend can check:

IP address

User agent / device fingerprint

If a token suddenly comes from another device/location → block it.

🧨 Example: What happens if someone steals my JWT and sends a request?
❌ If using only long-lived JWT:

Hacker can access until token expires → BAD DESIGN.

✔ If using short-lived JWT + refresh tokens:

Hacker gets ~10 min window only.

✔ If using refresh token rotation:

Once real user’s token rotates → hacker’s becomes invalid.

✔ If backend tracks refresh tokens:

You can instantly revoke the token.

🎯 How modern applications protect against stolen tokens
Mechanism	Protection
Access token lifespan = 5–15 minutes	Limits hacker usage window
Refresh token stored in HttpOnly Cookie	Prevents XSS theft
Refresh token rotation	Blocks replay attacks
Revoke token in DB	Immediately invalidate stolen tokens
HTTPS	Prevents network sniffing
IP/device verification	Detects unusual logins
⭐ Summary (Best Explanation)

If JWT is stolen → hacker can use it.
But real systems reduce the damage using:

Short-lived tokens

Secure HttpOnly cookies

Refresh token rotation

Token revocation in DB

HTTPS

Device/IP checks

So stolen token becomes useless very quickly.

5) JWT Utility / TokenService
@Service
public class JwtService {
    private final SecretKey key;
    private final Duration accessTokenValidity;
    private final Duration refreshTokenValidity;

    public JwtService(@Value("${jwt.secret}") String secret,
                      @Value("${jwt.access-expiration-minutes}") long accessMinutes,
                      @Value("${jwt.refresh-expiration-days}") long refreshDays) {
        this.key = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
        this.accessTokenValidity = Duration.ofMinutes(accessMinutes);
        this.refreshTokenValidity = Duration.ofDays(refreshDays);
    }

    public String generateAccessToken(User user) {
        Instant now = Instant.now();
        return Jwts.builder()
                .setSubject(user.getUsername())
                .claim("roles", user.getRoles())
                .setIssuedAt(Date.from(now))
                .setExpiration(Date.from(now.plus(accessTokenValidity)))
                .signWith(key, SignatureAlgorithm.HS256)
                .compact();
    }

    // For refresh tokens we typically use a random opaque token (not JWT), but you can also issue JWT refresh tokens.
    // Here we will generate a secure random string.
    public String generateRefreshTokenString() {
        return UUID.randomUUID().toString() + "-" + Base64.getUrlEncoder()
                .encodeToString(SecureRandom.getSeed(16));
    }

    public boolean validateAccessToken(String token) {
        try {
            Jwts.parserBuilder().setSigningKey(key).build().parseClaimsJws(token);
            return true;
        } catch (JwtException e) {
            return false;
        }
    }

    public String extractUsername(String token) {
        Claims claims = Jwts.parserBuilder().setSigningKey(key).build()
                .parseClaimsJws(token).getBody();
        return claims.getSubject();
    }

    public Instant getAccessTokenExpiry(String token) {
        Claims claims = Jwts.parserBuilder().setSigningKey(key).build()
                .parseClaimsJws(token).getBody();
        return claims.getExpiration().toInstant();
    }

    public Instant refreshTokenExpiryInstant() {
        return Instant.now().plus(refreshTokenValidity);
    }
}


Note: refresh tokens are stored in DB as opaque tokens (not JWT) — simpler to revoke and rotate.

6) Authentication Controller (login / refresh / logout)
@RestController
@RequestMapping("/auth")
public class AuthController {

    private final AuthenticationManager authenticationManager; // configure bean or use DaoAuthenticationProvider
    private final UserRepository userRepo;
    private final RefreshTokenRepository refreshRepo;
    private final JwtService jwtService;
    private final PasswordEncoder passwordEncoder;

    public AuthController(AuthenticationManager am, UserRepository ur,
                          RefreshTokenRepository rr, JwtService js, PasswordEncoder pe) {
        this.authenticationManager = am;
        this.userRepo = ur;
        this.refreshRepo = rr;
        this.jwtService = js;
        this.passwordEncoder = pe;
    }

    record LoginRequest(String username, String password) {}

    @PostMapping("/login")
    public ResponseEntity<?> login(@RequestBody LoginRequest req, HttpServletResponse response) {
        // authenticate
        Authentication auth = authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(req.username(), req.password())
        );

        User user = userRepo.findByUsername(req.username()).orElseThrow();
        String accessToken = jwtService.generateAccessToken(user);

        // create refresh token entry
        String refreshString = jwtService.generateRefreshTokenString();
        RefreshToken rt = new RefreshToken();
        rt.setToken(refreshString);
        rt.setUser(user);
        rt.setExpiryDate(jwtService.refreshTokenExpiryInstant());
        rt.setDeviceInfo("login-from-unknown"); // attach user-agent/ip
        refreshRepo.save(rt);

        // set HttpOnly cookie for refresh token
        Cookie cookie = new Cookie("refreshToken", refreshString);
        cookie.setHttpOnly(true);
        cookie.setSecure(true); // set true in prod (HTTPS)
        cookie.setPath("/"); 
        cookie.setMaxAge((int)Duration.between(Instant.now(), rt.getExpiryDate()).getSeconds());
        cookie.setSameSite("Lax"); // or Strict
        response.addCookie(cookie);

        Map<String, Object> resp = Map.of("accessToken", accessToken, "username", user.getUsername());
        return ResponseEntity.ok(resp);
    }

    @PostMapping("/refresh")
    public ResponseEntity<?> refresh(HttpServletRequest request, HttpServletResponse response) {
        // read refresh token cookie
        String refreshToken = extractRefreshTokenFromCookie(request);
        if (refreshToken == null) return ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();

        var optional = refreshRepo.findByToken(refreshToken);
        if (optional.isEmpty()) return ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();

        RefreshToken stored = optional.get();
        if (stored.getExpiryDate().isBefore(Instant.now())) {
            refreshRepo.delete(stored);
            return ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();
        }

        // rotate: remove old token & create new one
        refreshRepo.delete(stored);

        User user = stored.getUser();
        String newRefreshString = jwtService.generateRefreshTokenString();
        RefreshToken newRt = new RefreshToken();
        newRt.setToken(newRefreshString);
        newRt.setUser(user);
        newRt.setExpiryDate(jwtService.refreshTokenExpiryInstant());
        newRt.setDeviceInfo(stored.getDeviceInfo());
        refreshRepo.save(newRt);

        // set new cookie
        Cookie cookie = new Cookie("refreshToken", newRefreshString);
        cookie.setHttpOnly(true);
        cookie.setSecure(true);
        cookie.setPath("/");
        cookie.setMaxAge((int)Duration.between(Instant.now(), newRt.getExpiryDate()).getSeconds());
        cookie.setSameSite("Lax");
        response.addCookie(cookie);

        // new access token
        String newAccessToken = jwtService.generateAccessToken(user);
        return ResponseEntity.ok(Map.of("accessToken", newAccessToken));
    }

    @PostMapping("/logout")
    public ResponseEntity<?> logout(HttpServletRequest request, HttpServletResponse response) {
        String refreshToken = extractRefreshTokenFromCookie(request);
        if (refreshToken != null) {
            refreshRepo.findByToken(refreshToken).ifPresent(refreshRepo::delete);
        }
        // Remove cookie
        Cookie cookie = new Cookie("refreshToken", "");
        cookie.setHttpOnly(true);
        cookie.setSecure(true);
        cookie.setPath("/");
        cookie.setMaxAge(0);
        response.addCookie(cookie);
        return ResponseEntity.ok(Map.of("message", "logged out"));
    }

    private String extractRefreshTokenFromCookie(HttpServletRequest request) {
        if (request.getCookies() == null) return null;
        for (Cookie c : request.getCookies()) {
            if ("refreshToken".equals(c.getName())) return c.getValue();
        }
        return null;
    }
}

7) JWT Authentication Filter (validates access token on each request)
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    public JwtAuthenticationFilter(JwtService jwtService, UserDetailsService uds) {
        this.jwtService = jwtService;
        this.userDetailsService = uds;
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain chain)
            throws ServletException, IOException {

        final String authHeader = request.getHeader("Authorization");
        String jwt = null;
        String username = null;

        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            jwt = authHeader.substring(7);
            try {
                if (jwtService.validateAccessToken(jwt)) {
                    username = jwtService.extractUsername(jwt);
                }
            } catch (JwtException ex) {
                // invalid token — continue to 401 later if necessary
            }
        }

        if (username != null && SecurityContextHolder.getContext().getAuthentication() == null) {
            UserDetails ud = userDetailsService.loadUserByUsername(username);
            UsernamePasswordAuthenticationToken authToken =
                new UsernamePasswordAuthenticationToken(ud, null, ud.getAuthorities());
            authToken.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
            SecurityContextHolder.getContext().setAuthentication(authToken);
        }

        chain.doFilter(request, response);
    }
}

        http
            .csrf().disable() // if using cookies for refresh, consider CSRF protection for state-changing endpoints
            .sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            .and()
            .authorizeHttpRequests()
                .requestMatchers("/auth/**").permitAll()
                .anyRequest().authenticated()
            .and()
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    @Bean
    public AuthenticationManager authenticationManager(AuthenticationConfiguration config) throws Exception {
        return config.getAuthenticationManager();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

                .build();
    }

10) Example usage & flow recap

Login (/auth/login)

Client posts username/password → server authenticates → issues short-lived access token and sets refresh token cookie (HttpOnly).

API requests

Client attaches Authorization: Bearer <access-token> on each request.

Server validates token in JwtAuthenticationFilter on every request.

Access token expires

Client calls /auth/refresh (no body). The browser automatically sends the HttpOnly cookie with refresh token.

Server validates stored refresh token, rotates it (delete old → create new), sets new refresh cookie, and returns a fresh access token.

Logout or suspicious activity

Call /auth/logout. Server deletes refresh token in DB and clears cookie. This revokes future refresh attempts (and because access tokens are short-lived, full access will end quickly).

11) Important Security Notes (production)

HTTPS mandatory — cookie & token must be transmitted securely.

Use strong random secrets; rotate them if needed.

Consider device / IP / fingerprinting for extra checks.

Rate limit refresh attempts and login attempts to avoid brute force.

Use CSRF protection if you store session-like state in cookies; for refresh endpoint a same-site cookie + CSRF token helps.

Audit & logging suspicious refresh/login attempts.

Limit scope of tokens (use scopes/claims to minimize damage).

Rotate refresh tokens on every refresh (we implemented this).

Invalidate all refresh tokens on password change.

Handling shared values in multithreading is one of the most important topics in Java because multiple threads accessing the same data can cause race conditions, data inconsistency, and visibility issues.

Below is a clear, interview-oriented explanation with examples 👇

1️⃣ The Problem: Shared Mutable State

When multiple threads read/write the same variable, issues occur.

Example (Problematic)
class Counter {
    int count = 0;

    void increment() {
        count++; // NOT thread-safe
    }
}


If 2 threads call increment() simultaneously:

Both may read count = 0

Both write back count = 1
❌ Expected = 2, Actual = 1

2️⃣ Solutions to Handle Shared Values
✅ 1. synchronized Keyword (Most Common)

Ensures only one thread at a time can access the shared resource.

Method-level synchronization
class Counter {
    private int count = 0;

    synchronized void increment() {
        count++;
    }

    synchronized int getCount() {
        return count;
    }
}


✔ Thread-safe
❌ Slower due to locking

✅ 2. synchronized Block (Preferred for Performance)

Locks only the critical section, not the whole method.

class Counter {
    private int count = 0;

    void increment() {
        synchronized (this) {
            count++;
        }
    }
}


✔ Better performance
✔ Fine-grained locking

✅ 3. volatile (For Visibility, NOT Atomicity)

Ensures latest value is visible to all threads.

class Shared {
    volatile boolean running = true;
}


⚠️ volatile does NOT prevent race conditions

❌ Not safe for:

volatile int count; // count++ still unsafe


✔ Use only when:

One thread writes

Others only read

✅ 4. Atomic Variables (java.util.concurrent.atomic)

Best for counters and simple shared values.

import java.util.concurrent.atomic.AtomicInteger;

class Counter {
    AtomicInteger count = new AtomicInteger(0);

    void increment() {
        count.incrementAndGet();
    }
}


✔ Lock-free
✔ High performance
✔ Thread-safe

✅ 5. Locks (ReentrantLock) – Advanced Control

More flexible than synchronized.

import java.util.concurrent.locks.ReentrantLock;

class Counter {
    private int count = 0;
    private final ReentrantLock lock = new ReentrantLock();

    void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }
}


✔ Try-lock, fairness, interruptible
❌ More code

✅ 6. Thread-Local Variables (ThreadLocal)

Each thread gets its own copy (no sharing at all).

ThreadLocal<Integer> threadLocal =
        ThreadLocal.withInitial(() -> 0);

threadLocal.set(threadLocal.get() + 1);


✔ No synchronization needed
✔ Used in Spring (SecurityContext, Transactions)

✅ 7. Immutable Objects (Best Design Approach)

If data never changes, it’s automatically thread-safe.

final class User {
    private final String name;

    User(String name) {
        this.name = name;
    }

    public String getName() {
        return name;
    }
}


✔ Safest
✔ Recommended wherever possible

3️⃣ When to Use What? (Interview Gold)
Scenario	Best Option
Simple counter	AtomicInteger
Complex logic	synchronized / ReentrantLock
Flag / config	volatile
Per-thread data	ThreadLocal
Read-heavy data	Immutable objects
4️⃣ Common Interview Question

❓ Does volatile make code thread-safe?
👉 NO – It only guarantees visibility, not atomicity.

5️⃣ One-Line Interview Answer

“Shared values in multithreading are handled using synchronization mechanisms like synchronized, locks, atomic variables, volatile for visibility, thread-local storage for isolation, and immutable objects to avoid shared mutable state.”

Here are the most common Semaphore interview questions (with clear, interview-ready answers) tailored for Java / backend developer interviews (3+ years).

1️⃣ What is a Semaphore?

Answer:
A Semaphore is a concurrency control mechanism used to limit access to a shared resource by multiple threads.
It maintains a set of permits. A thread must acquire a permit before accessing the resource and release it after use.

In Java, it is provided by:

java.util.concurrent.Semaphore

2️⃣ Why do we need Semaphore when we already have synchronized?

Answer:

| synchronized             | Semaphore              |
| ------------------------ | ---------------------- |
| Allows **only 1 thread** | Allows **N threads**   |
| Implicit lock            | Explicit permits       |
| No fairness option       | Supports fairness      |
| No tryAcquire            | tryAcquire() available |


👉 Semaphore is used when we want to allow limited concurrent access, not just one thread.

3️⃣ Types of Semaphores

Answer:

🔹 Binary Semaphore

Has 1 permit

Works like a mutex/lock

Semaphore sem = new Semaphore(1);

🔹 Counting Semaphore

Has multiple permits

Allows multiple threads

Semaphore sem = new Semaphore(3);

4️⃣ How Semaphore works internally?

Answer:

acquire() → decreases permit count

release() → increases permit count

If permits = 0 → thread blocks

When permit released → waiting thread resumes

5️⃣ Simple Java Example (Interview Favorite)
Semaphore semaphore = new Semaphore(2); // 2 threads allowed

public void accessResource() {
    try {
        semaphore.acquire();
        System.out.println(Thread.currentThread().getName() + " acquired permit");
        Thread.sleep(1000);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        semaphore.release();
        System.out.println(Thread.currentThread().getName() + " released permit");
    }
}


💡 At most 2 threads can execute the critical section.

6️⃣ What is Fairness in Semaphore?

Answer:
Fair semaphore ensures FIFO order for threads waiting for permits.

Semaphore semaphore = new Semaphore(2, true);


true → Fair

false → Non-fair (default, better performance)

7️⃣ Difference between Semaphore and Mutex / Lock
| Feature   | Semaphore       | Mutex / Lock     |
| --------- | --------------- | ---------------- |
| Permits   | Multiple        | One              |
| Ownership | No owner thread | Owner thread     |
| Release   | Any thread      | Only owner       |
| Use case  | Resource pool   | Critical section |

8️⃣ What happens if a thread releases without acquiring?

Answer:
Semaphore does not track ownership.
So releasing without acquiring increases permits, which may cause bugs.

👉 This is why Locks are safer for mutual exclusion.

9️⃣ Semaphore vs CountDownLatch vs CyclicBarrier
| Feature      | Semaphore    | CountDownLatch | CyclicBarrier       |
| ------------ | ------------ | -------------- | ------------------- |
| Purpose      | Limit access | Wait for tasks | Synchronize threads |
| Reusable     | Yes          | No             | Yes                 |
| Permit based | Yes          | No             | No                  |

🔟 Real-World Use Cases (Important in Interviews)

✅ Database connection pool
✅ Thread pool limiting
✅ Rate limiting
✅ API throttling
✅ Printer / resource sharing

1️⃣1️⃣ Can Semaphore cause deadlock?

Answer:
Yes, if permits are not released or acquired in the wrong order.

Example:

semaphore.acquire();
// exception occurs
// release not called → deadlock


✔ Always release in finally block.

1️⃣2️⃣ tryAcquire() vs acquire()
if (semaphore.tryAcquire()) {
    // access resource
    semaphore.release();
}

| acquire()          | tryAcquire()        |
| ------------------ | ------------------- |
| Blocks             | Non-blocking        |
| Waits indefinitely | Returns immediately |

1️⃣3️⃣ One-line Interview Answer (Strong)

“Semaphore is used to control concurrent access to a shared resource by maintaining permits. Unlike synchronized, it allows multiple threads and supports fairness and non-blocking access.”

🔥 Quick Interview Tip

If interviewer asks:

“Where did you use Semaphore in your project?”

Say:

“We used Semaphore to limit concurrent API calls / database connections to avoid resource exhaustion.”

In Java, exceptions can be thrown from methods in two main ways:

1️⃣ Throwing an exception using throw

Use throw inside the method body to explicitly throw an exception.

✅ Example
public void validateAge(int age) {

📌 throw is used to actually throw an exception object.

Use throws in the method signature to declare exceptions that the method may pass to the caller.

✅ Example
public void readFile() throws IOException {

📌 throws is used to declare exceptions, not throw them.

        throw new Exception("Invalid withdrawal amount");

Must be handled or declared

Extend Exception (except RuntimeException)

Extend RuntimeException

No need to declare using throws

Abstraction in Java means hiding implementation details and showing only essential features to the user.
Java achieves abstraction mainly in two ways:

1️⃣ Abstraction using Abstract Classes

An abstract class is a class declared with the abstract keyword.

🔹 Key points

Can have abstract methods (without body)

Can have concrete methods (with body)

Can have fields, constructors

A class can extend only one abstract class

✅ Example
abstract class Vehicle {
    abstract void start();   // abstract method

    void fuelType() {        // concrete method
        System.out.println("Uses fuel");
    }
}

class Car extends Vehicle {
    @Override
    void start() {
        System.out.println("Car starts with key");
    }
}

🔹 Usage
Vehicle v = new Car();
v.start();       // Car starts with key
v.fuelType();   // Uses fuel


📌 What’s abstracted?
User knows Vehicle can start, but how it starts is hidden.

2️⃣ Abstraction using Interfaces (Most Important)

An interface is a blueprint of a class.

🔹 Key points

Methods are public & abstract by default

Supports multiple inheritance

Variables are public static final

From Java 8:

default methods allowed

static methods allowed

✅ Example
interface Payment {
    void pay();   // abstract method
}

class CreditCardPayment implements Payment {
    public void pay() {
        System.out.println("Payment through Credit Card");
    }
}

class UPIPayment implements Payment {
    public void pay() {
        System.out.println("Payment through UPI");
    }
}

🔹 Usage
Payment p = new UPIPayment();
p.pay();   // Payment through UPI


📌 What’s abstracted?
User only knows pay() exists — payment logic is hidden.

🔍 Abstract Class vs Interface (Quick Interview Table)
Feature	Abstract Class	Interface
Methods	Abstract + Concrete	Abstract (default/static allowed)
Variables	Instance variables	public static final
Inheritance	Single	Multiple
Constructors	Yes	No
Use case	IS-A + shared code	Contract / Capability
🎯 Real-Time Example (Interview Friendly)

Interface → What to do
Payment, Repository, Service

Abstract Class → What + How (partially)
AbstractService, BaseEntity

🧠 One-Line Interview Answer

Abstraction in Java is achieved using abstract classes and interfaces, which hide implementation details and expose only essential functionalities.

You want to store the pairs of elements whose sum = target (5) from the array.

Array:

int[] arr = {1, 3, 2, 4, 6, 8};
int target = 5;


Valid pairs:

(1, 4)

(3, 2)

Below are clean and interview-ready solutions, especially using Java Streams, since that’s what you usually prefer.

✅ Solution 1: Using Java Streams (Store pairs)
int[] arr = {1, 3, 2, 4, 6, 8};
int target = 5;

List<List<Integer>> pairs =
        IntStream.range(0, arr.length)
                .boxed()
                .flatMap(i ->
                        IntStream.range(i + 1, arr.length)
                                .filter(j -> arr[i] + arr[j] == target)
                                .mapToObj(j -> List.of(arr[i], arr[j]))
                )
                .toList();

System.out.println(pairs);

🖨 Output
[[1, 4], [3, 2]]

✅ Solution 2: Using Set (Optimized – O(n))

This is more efficient and commonly appreciated in interviews.

int[] arr = {1, 3, 2, 4, 6, 8};
int target = 5;

Set<Integer> seen = new HashSet<>();
List<List<Integer>> pairs = new ArrayList<>();

for (int num : arr) {
    int complement = target - num;           //4, 2, 3
    if (seen.contains(complement)) {
        pairs.add(List.of(complement, num)); //(3,2)
    }
    seen.add(num);                           //1, 3
}

System.out.println(pairs);

🖨 Output

STRING BUFFER:-
----------------

Good question 👍 — this is a classic Java interview favorite.

Why StringBuffer is thread-safe

StringBuffer is thread-safe because all its public methods are synchronized.

That’s it. That’s the core reason.

What does “synchronized” mean here?

When a method is marked synchronized, only one thread can execute that method on the same object at a time.

So if two threads try to modify the same StringBuffer instance:

Thread A enters append()

Thread B must wait until Thread A finishes

This prevents data corruption

Example (simplified)
StringBuffer sb = new StringBuffer();

sb.append("Hello");
sb.append(" World");


Behind the scenes, methods look like this:

public synchronized StringBuffer append(String str) {
    // modify internal character array safely
}


Because of synchronized, Java automatically:

Acquires a lock on the StringBuffer object

Releases it when the method finishes

What problem does this prevent?

Without synchronization:

Two threads could update the internal char array at the same time

Result → inconsistent or corrupted string

With StringBuffer:

Operations are atomic

Internal state is always consistent

Comparison with StringBuilder
| Feature         | StringBuffer       | StringBuilder   |
| --------------- | ------------------ | --------------- |
| Thread-safe     | ✅ Yes              | ❌ No            |
| Synchronization | Yes (method-level) | No              |
| Performance     | Slower             | Faster          |
| Use case        | Multi-threaded     | Single-threaded |


👉 If multiple threads modify the same string, use StringBuffer
👉 If single thread, use StringBuilder (recommended in most cases)

Important interview note ⚠️

StringBuffer is thread-safe, but not always the best choice

Method-level synchronization is heavy

In modern Java:

Prefer StringBuilder

Use explicit synchronization only when needed

One-liner interview answer 💡

StringBuffer is thread-safe because all its public methods are synchronized, ensuring only one thread can modify the object at a time.


STRING BUILDER:-
----------------

Short answer first 👇
StringBuilder is NOT thread-safe.

Now the why 👇

Why StringBuilder is not thread-safe

Unlike StringBuffer, StringBuilder methods are NOT synchronized.

That means:

Multiple threads can access the same StringBuilder object at the same time

If they modify it concurrently → race condition + data corruption

Example problem
StringBuilder sb = new StringBuilder();

Thread t1 = new Thread(() -> sb.append("A"));
Thread t2 = new Thread(() -> sb.append("B"));

t1.start();
t2.start();


Possible outputs:

AB

BA

Corrupted result (in rare cases)

Why?
Because append() is not atomic in StringBuilder.

How StringBuilder looks internally
public StringBuilder append(String str) {
    // no synchronized keyword
}


So:

No locking

No thread coordination

Faster execution

Why did Java introduce StringBuilder then?

Because 90% of string operations are single-threaded.

So Java needed:

A faster alternative to StringBuffer

Same API

No synchronization overhead

👉 That’s StringBuilder.

Can we make StringBuilder thread-safe?

Yes, externally.

Option 1: synchronize manually
synchronized (sb) {
    sb.append("Hello");
}

Option 2: use StringBuffer
StringBuffer sb = new StringBuffer();

Option 3 (recommended modern approach)
String result = String.join("", listOfStrings);

Interview-friendly comparison 👇
| Feature         | StringBuilder | StringBuffer   |
| --------------- | ------------- | -------------- |
| Thread-safe     | ❌ No          | ✅ Yes          |
| Synchronization | ❌ None        | ✅ Method-level |
| Performance     | 🚀 Faster     | 🐢 Slower      |
| Introduced in   | Java 5        | Java 1.0       |

One-liner interview answer 💡

StringBuilder is not thread-safe because its methods are not synchronized, allowing concurrent modification by multiple threads.

Here are the key functions (methods) of the Java Stream API with their definitions:
Intermediate Operations:-
-------------------------
Intermediate operations transform a stream into another stream and are lazy, meaning they are not executed until a terminal operation is invoked. 

🗸 filter(Predicate predicate): Returns a stream consisting of elements that match the given predicate (a boolean-valued function).

🗸 map(Function mapper): Returns a stream consisting of the results of applying the given function to each element of the source stream, transforming the elements.

🗸 flatMap(Function mapper): Replaces each element with the contents of a mapped stream produced by the mapping function and flattens all the resulting streams into a single stream.

🗸 distinct(): Returns a stream consisting of the unique elements of the source stream, based on their equals() method.

🗸 sorted(): Returns a stream with elements sorted according to their natural order (if they implement Comparable).

🗸 sorted(Comparator comparator): Returns a stream sorted according to the provided Comparator.

🗸 peek(Consumer action): Performs an action on each element as it passes through the pipeline without modifying the elements, useful for debugging or logging.

🗸 limit(long maxSize): Truncates the stream to contain no more than the specified number of elements.

🗸 skip(long n): Discards the first n elements of the stream and returns a stream with the remaining elements.

🗸 takeWhile(Predicate predicate) (Java 9+): Consumes elements as long as the predicate is true, then stops.

🗸 dropWhile(Predicate predicate) (Java 9+): Skips elements while the predicate is true, then includes all remaining elements. 

Terminal Operations:-
---------------------
Terminal operations initiate the stream pipeline processing and produce a non-stream result (a value, a collection, or a side effect). Once a terminal operation is performed, 
the stream is consumed and cannot be used again. 

🗸 forEach(Consumer action): Performs an action for each element of the stream.

🗸 collect(Collector collector): Accumulates the elements into a mutable result container, such as a List, Set, or Map, using a Collector.

🗸 reduce(initialValue, BinaryOperator accumulator): Combines the elements of the stream into a single result using an associative accumulation function.

🗸 count(): Returns the number of elements in the stream.

🗸 min(Comparator comparator) / max(Comparator comparator): Returns the minimum or maximum element in the stream according to the provided Comparator in an Optional.

🗸 anyMatch(Predicate predicate): Returns whether at least one element matches the provided predicate.

🗸 allMatch(Predicate predicate): Returns whether all elements match the provided predicate.

🗸 noneMatch(Predicate predicate): Returns whether no elements match the provided predicate.

🗸 findFirst(): Returns an Optional describing the first element of the stream, if the stream has a defined encounter order.

🗸 findAny(): Returns an Optional describing any element of the stream; useful in parallel operations for better performance.

🗸 toArray(): Returns an array containing the elements of the stream.

How to save multiple data at once in spring boot:

Good question 👍 — because many developers misunderstand this.

👉 Yes, saveAll() often still generates multiple INSERT statements, even though it looks like a bulk operation.

If you truly want insert at once (single DB round trip / grouped statement), you need to go beyond default JPA behavior.

Let’s break it down clearly.

🔥 Option 1 — TRUE Batch Insert (Most Recommended)

Hibernate can send multiple inserts in one network call using JDBC batching.

✅ Step 1 — Enable batching
spring.jpa.properties.hibernate.jdbc.batch_size=50
spring.jpa.properties.hibernate.order_inserts=true
spring.jpa.properties.hibernate.order_updates=true

Now instead of:

INSERT INTO user...
INSERT INTO user...
INSERT INTO user...

Hibernate groups them like:

Batch of 50 → single DB trip
Batch of 50 → single DB trip

👉 This is how high-performance production apps work.

⚠️ Without this property → saveAll() behaves almost like a loop.

✅ VERY IMPORTANT (Most People Miss This)

If your ID is auto-generated like this:

@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)

🚨 Batching will NOT work properly because the DB must return each generated ID immediately.

👉 Use:

@GeneratedValue(strategy = GenerationType.SEQUENCE)

(or TABLE strategy)

✔ Now batching becomes effective.

This alone can improve insert speed by 5–10x.


If interviewer says:

👉 “saveAll still inserts one by one. How do you fix it?”

Say:

“I enable Hibernate JDBC batching and avoid IDENTITY generation. For extreme performance, I use JdbcTemplate batch updates.”

That answer sounds very experienced.

In Spring Boot, saving multiple records at once is called Batch Insert or Bulk Save. It is very important for performance because inserting one-by-one causes many DB round trips.

Let’s go from most commonly used → advanced production approach.

✅ 1. Use saveAll() (Easiest & Most Used)

If you are using Spring Data JPA:

Repository
public interface UserRepository extends JpaRepository<User, Long> {
}
Service
@Transactional
public void saveUsers(List<User> users){
    userRepository.saveAll(users);
}

👉 That’s it.
Spring will insert all records in one operation (internally multiple inserts, but optimized).

✔ Clean
✔ Production-safe
✔ Interview-friendly

⭐ Most companies use this unless handling very large data.

✅ 2. Enable TRUE Batch Inserts (VERY IMPORTANT ⚡)

By default, Hibernate may still fire separate insert statements.

Enable batching in application.properties:

spring.jpa.properties.hibernate.jdbc.batch_size=50
spring.jpa.properties.hibernate.order_inserts=true
spring.jpa.properties.hibernate.generate_statistics=true

👉 Now instead of:

insert...
insert...
insert...

It sends grouped batches to DB → Huge performance gain.

✔ Recommended batch size: 30–50

(Bigger is NOT always better — can cause memory issues.)

✅ 3. Wrap in Transaction (CRITICAL)
@Transactional
public void saveUsers(List<User> users){
    userRepository.saveAll(users);
}

Why?

👉 If record #45 fails:

✔ Everything rolls back
✔ No partial data

Without transaction → inconsistent DB.

✅ 4. For VERY Large Data (Senior-Level Approach)

If inserting lakhs/millions of rows:

Use EntityManager.

@PersistenceContext
private EntityManager entityManager;


@Transactional
public void batchInsert(List<User> users){


    int batchSize = 50;


    for(int i=0; i<users.size(); i++){
        entityManager.persist(users.get(i));


        if(i % batchSize == 0){
            entityManager.flush();
            entityManager.clear();
        }
    }
}

👉 Prevents memory overflow inside Hibernate.

✔ Used in data migration
✔ Used in ETL jobs
✔ Used by senior engineers

🔥 Performance Comparison
Method	Speed	When to Use
save() loop	❌ Slow	Never for bulk
saveAll()	✅ Good	90% of apps
Hibernate batch	🔥 Very fast	High traffic
EntityManager	🚀 Extreme	Massive datasets
❗Common Mistake Developers Make

🚨 Doing this:

for(User u : users){
   userRepository.save(u); // BAD
}

👉 Causes N DB calls.

Always prefer:

✅ saveAll()

⭐ Interview Answer (Golden)

If interviewer asks:

👉 “How do you insert multiple records in Spring Boot efficiently?”

Say:

“I use saveAll() from Spring Data JPA with Hibernate batch enabled and wrap it inside a transaction. For very large datasets, I switch to EntityManager batching to avoid memory issues.”

This sounds mid-to-senior level instantly.

⚠️ MOST IMPORTANT STEP (People Miss This)

👉 Database must return ID immediately → breaks batching.

✔ Much faster inserts

✅ Step 2 — Always Insert Inside a Transaction

What is REST API:-
-----------------

✅ 30–40 Second Interview Answer (Best Version)

REST API stands for Representational State Transfer Application Programming Interface. It is an architectural style used to build scalable web services that communicate over HTTP.
In REST, everything is treated as a resource and is accessed using standard HTTP methods like GET, POST, PUT, PATCH, and DELETE. REST APIs are stateless, meaning each request contains all the information needed for the server to process it.
They typically exchange data in JSON format, making them lightweight and fast. In my projects, I have used REST APIs in Spring Boot to enable communication between frontend and backend systems and to support microservice interactions.

✅ If Interviewer Says: “Explain in Simple Terms”

A REST API is a way for two applications to communicate over the internet using HTTP protocols. For example, when a user searches for products in an app, the frontend sends a GET request to the REST API, the server processes it, fetches data from the database, and returns it in JSON format.

✅ If Interviewer Wants More Depth (1–2 Minute Answer)

REST is an architectural style that follows principles such as stateless communication, client-server separation, cacheability, and a uniform interface.
Each resource is identified by a URL, and actions are performed using HTTP methods. For example, GET retrieves data, POST creates a resource, PUT updates it, and DELETE removes it.
One major advantage of REST is scalability because the server does not store client context. It is also lightweight compared to SOAP since it commonly uses JSON.
In my experience working with Spring Boot, I have designed REST endpoints, handled request and response objects, implemented proper HTTP status codes, and ensured secure communication using authentication tokens.

What are the solid principles in java:-

SOLID is a set of 5 object-oriented design principles that help you write clean, maintainable, scalable, and flexible Java code.

SOLID =
👉 S — Single Responsibility Principle
👉 O — Open/Closed Principle
👉 L — Liskov Substitution Principle
👉 I — Interface Segregation Principle
👉 D — Dependency Inversion Principle

Let’s understand each with simple Java examples 👇

✅ 1. Single Responsibility Principle (SRP)

Definition:
A class should have only one reason to change (one responsibility).

❌ Wrong Example

class Invoice {
    void calculateTotal() { }
    void printInvoice() { }
    void saveToDatabase() { }
}

This class handles:

business logic

printing

database work

✅ Correct Approach

class Invoice {
    void calculateTotal() { }
}

class InvoicePrinter {
    void print(Invoice invoice) { }
}

class InvoiceRepository {
    void save(Invoice invoice) { }
}

👉 Each class has one job.

####################################################

✅ 2. Open/Closed Principle (OCP)

Definition:
Software should be open for extension but closed for modification.

You should add new behavior without changing existing code.

❌ Wrong

class PaymentService {
    void pay(String type) {
        if(type.equals("UPI")) { }
        else if(type.equals("CARD")) { }
    }
}

Every new payment → modify class.

✅ Correct

interface Payment {
    void pay();
}

class UpiPayment implements Payment {
    public void pay() { }
}

class CardPayment implements Payment {
    public void pay() { }
}

Now just add new class → no modification needed.

####################################################

✅ 3. Liskov Substitution Principle (LSP)

Definition:
A subclass should be able to replace its parent class without breaking behavior.

❌ Wrong

class Bird {
    void fly() {}
}

class Ostrich extends Bird {
    void fly() {
        throw new UnsupportedOperationException();
    }
}

Ostrich cannot fly → violates LSP.

✅ Correct

interface Bird {}

interface FlyingBird {
    void fly();
}

class Sparrow implements Bird, FlyingBird {
    public void fly() {}
}

class Ostrich implements Bird {}

####################################################

✅ 4. Interface Segregation Principle (ISP)

Definition:
Clients should not be forced to depend on methods they don’t use.

❌ Wrong

interface Worker {
    void work();
    void eat();
}

class Robot implements Worker {
    public void work() {}
    public void eat() {} // unnecessary
}

✅ Correct

interface Workable {
    void work();
}

interface Eatable {
    void eat();
}

Now classes implement only what they need.

####################################################

✅ 5. Dependency Inversion Principle (DIP)

Definition:
High-level modules should depend on abstractions (Interfaces), not concrete classes.

❌ Wrong

class MySQLDatabase { }

class UserService {
    MySQLDatabase db = new MySQLDatabase();
}

Tightly coupled.

✅ Correct

interface Database {
    void save();
}

class MySQLDatabase implements Database {
    public void save() {}
}

class UserService {
    private Database db;

    UserService(Database db) {
        this.db = db;
    }
}

Now you can switch DB easily.

🎯 Easy Memory Trick

👉 S — One class → One job
👉 O — Extend without changing
👉 L — Child behaves like parent
👉 I — Small specific interfaces
👉 D — Depend on interfaces

🚀 Why SOLID matters (especially in Spring Boot)

You’ll see SOLID everywhere in:

Spring Dependency Injection

Service & Repository layers

Microservices

Clean Architecture

Testable code

You already know:

class name

methods

fields

Reflection (Runtime)
Class<?> clazz = Class.forName("User");

Here the class is loaded dynamically.

✅ What Reflection Can Do

Using reflection you can:

✔ Get class name
✔ Get methods
✔ Access private fields
✔ Call methods dynamically

✅ Easy Example
Sample class

✅ Interface vs Abstraction (Abstract Class)

Interfaces

Car can move

How hashmap works internally - Hashing technique - How it will resolve collisions of hashing:-

This is a very important Java interview question 👍
Let’s explain HashMap internal working step-by-step in a simple way.

✅ How HashMap Works Internally

A HashMap stores data as key–value pairs.

HashMap<Integer, String> map = new HashMap<>();
map.put(101, "A");

Internally HashMap uses:

👉 Array of Nodes (Buckets)

🔹 Step 1 — Hash Calculation

When you insert a key:

map.put(key, value);

Java calculates:

hash = key.hashCode();

Then index is calculated:

index = hash & (arraySize - 1)

👉 This decides which bucket to store data.

🔹 Step 2 — Store in Bucket

Each array position is called a bucket.

Internally structure looks like:

Array
[0] -> null
[1] -> Node(key,value)
[2] -> Node -> Node -> Node
[3] -> null

Each entry is stored as a:

static class Node<K,V> {
    final int hash;
    final K key;
    V value;
    Node<K,V> next;
}

👉 So datatype used = Array + LinkedList (initially)

✅ What is Collision?

Collision happens when:

👉 Two different keys produce the same bucket index.

Example:

Key1 → index 2
Key2 → index 2

Both want same location.

✅ How HashMap Resolves Collisions
✅ Java 7 (Old)

Collision handled using:

👉 Linked List

Bucket[2]
   ↓
Node1 → Node2 → Node3

New node added to chain.

✅ Java 8+ (Important Interview Point)

If collisions become too many:

👉 LinkedList converts into Red-Black Tree

Condition:

Bucket size ≥ 8

Array size ≥ 64

Before:
Node → Node → Node → Node

After:
     Tree Structure
        (Balanced)
🔥 Why Tree?
Structure	Search Time
LinkedList	O(n)
Red-Black Tree	O(log n)

Improves performance.

✅ Retrieval (get operation)
map.get(key);

Steps:

Calculate hashCode

Find bucket index

Traverse linked list / tree

Compare using equals()

⚠️ Important:

hashCode() → finds bucket

equals() → finds exact key

🎯 Datatypes Used Internally

👉 Array + LinkedList + Red-Black Tree (Java 8+)

🧠 Interview 30-Second Answer

HashMap internally uses an array of buckets where each bucket stores nodes. The index is calculated using the key’s hashCode. When multiple keys map to the same bucket, collision occurs and is handled using a LinkedList, which converts into a Red-Black Tree in Java 8 when entries exceed a threshold, improving search performance.
